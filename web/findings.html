DesignGym 2.0 — Honest Findings

Post-deployment analysis. No spin. What worked, what didn't, and what to do next.

The Deployment Honesty Bug

The original Space README claimed results from fine-tuned SFT/GRPO LoRA adapters, but the deployed HF Space was silently using base Qwen via the HF Inference Router. The Router only hosts a curated model catalogue and cannot serve user LoRA adapters. Visitors saw "SFT-LLM Picker" in the UI, but every inference call hit the unmodified base model.

This fix loads the actual LoRA adapters locally with PEFT inside the Space worker process. Every step now shows an honest label: finetuned_sft, finetuned_grpo, local_base, or heuristic.

What Worked

SFT: 0% → 100% valid JSON

This is the biggest single win. Base Qwen 0.5B cannot produce the {"choice": N} action format at all. After SFT it does so reliably. This is a capability phase transition, not a marginal improvement.

GRPO: Highest per-step reward

GRPO accumulates 1.854 avg total reward — 17% more than heuristic (1.588). It picks bolder actions that move metrics more per step. On the hardest task (dense_flyer), GRPO's instruction score (0.687) beats every other backend by a wide margin.

100% LLM steer rate

The parse fix (handling bare integers + regex fallback) means every single step is now model-driven. Zero fallback to heuristic candidate selection.

What Didn't Work (Yet)

Heuristic still wins on final score

The hand-coded heuristic gets 0.738 avg final score vs SFT's 0.702 and GRPO's 0.694. This is expected: the heuristic was written with full knowledge of the reward function and metric weights. It's a strong ceiling for a small model to beat.

Small SFT vs Base lift

SFT beats base by only +0.005 on final score. The adapter teaches format compliance (0% → 100% valid JSON) but doesn't dramatically change which actions the model prefers. This suggests the SFT training data (heuristic trajectories) could be enriched with more diverse, higher-quality demonstrations.

GRPO needs more training

The current GRPO run was ~200 optimization steps on a free Colab T4. Published GRPO results on comparable tasks use 1000-5000 steps with best-of-N=8 sampling. The infrastructure and reward signal are proven — it just needs more compute.

Why Does GRPO Have High Reward But Lower Final Score?

This is the most interesting finding. GRPO maximizes per-step reward (delta-based shaping), but the final score depends on the endpoint state — not the sum of deltas. GRPO picks aggressive actions that generate large positive deltas, but sometimes overshoots or doesn't converge cleanly to a high final state.

The reward function rewards improvement. The final score rewards position. These correlate but don't perfectly align — especially with limited training budget.

Fix: add a terminal bonus proportional to final score, or weight the last step's reward more heavily.

The 0.5B Elephant in the Room

Qwen 2.5 0.5B has 494M parameters. The observation prompt is ~500 tokens of structured layout state + metrics. For a model this small, the "understand complex multi-metric state and reason about which of 6 actions best addresses the weakest metric" task is near the edge of capability.

Evidence: the model often returns the correct format (valid JSON) but picks somewhat random choices — it hasn't internalized the relationship between worst_metrics and action types deeply enough. A 3B model with the same LoRA adapter would likely show a much bigger adapter-vs-base gap.

Concrete Next Steps

  • Scale GRPO training: 1000+ steps, best-of-N=8, gradient accumulation. A100/H100 for a few hours, or HF training jobs.
  • Try Qwen 3B: Same LoRA rank (16), same data. 4-bit quantization fits in 16GB. The reasoning capacity jump should be significant.
  • Process reward model: Train a small critic network that scores partial trajectories, giving GRPO per-step signal instead of episode-end only.
  • Curriculum: Start GRPO on poster_basic_v1 (easy) until convergence, then add editorial and dense_flyer. Currently all tasks train simultaneously.
  • Reward terminal bonus: Add 0.2 * final_score to the last step's reward so GRPO optimizes for endpoint quality, not just per-step deltas.
  • Richer SFT data: Include multiple heuristic variations and best-of-N filtered trajectories as SFT training data instead of single-run heuristic traces.
